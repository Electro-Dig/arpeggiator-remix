# WAIC 拍照分享海报设计规范

## 目标

在录音确认之后加入一次明确、可恢复的拍照流程，并把参与者照片合成到最终分享海报中。最终海报与录音一同在手机分享页提供下载，使结果从“下载操作页”升级为具有本人记忆点的现场作品。

## 已选方案

采用“现场单曲封面 + 演奏指纹”的混合方向。

- 不采用纯照片亭方案：个人感强，但会削弱双手音乐项目的识别。
- 不采用纯数据卡方案：结构清楚，但仍然缺少参与者本人。
- 不采用重度游戏角色卡：趣味性高，但与当前克制、极简的界面语气不一致。
- 选定方案以人物照片为主视觉，用网格、双手轨迹、节奏点和编号形成项目专属的演奏指纹。

## 用户流程

1. 录音停止后进入录音确认页。
2. 点击“确认并拍照”。
3. 页面进入拍照倒计时，依次显示 3、2、1；使用现有手势摄像头，不发起第二次摄像头授权。
4. 倒计时结束后从摄像头视频捕获一帧。
5. 显示照片确认页，提供：
   - “使用这张照片”：进入上传与海报生成。
   - “重新拍摄”：重新开始 3、2、1。
   - “无照片继续”：使用项目生成式背景作为封面，保证摄像头异常、参与者不愿拍照或现场拥挤时仍能完成流程。
6. 上传录音并取得分享 token、打卡编号和分享 URL。
7. 在浏览器本地把照片、分享 URL、二维码、打卡编号和项目图形合成为最终海报。
8. 只上传最终海报文件，不上传原始照片。
9. 海报上传成功后进入最终分享页；现场电脑显示二维码和海报预览，手机端可试听/下载录音并下载同一张海报。

## 状态模型

录音状态机新增以下阶段：

- `photo-countdown`：3 秒拍照倒计时。
- `photo-review`：显示捕获结果，等待使用、重拍或无照片继续。
- `uploading`：上传录音、渲染海报、上传海报。
- `shared`：显示最终海报与二维码。

关键转换：

- `review + PHOTO_REQUEST -> photo-countdown`
- `photo-countdown + PHOTO_CAPTURED -> photo-review`
- `photo-review + PHOTO_ACCEPTED -> uploading`
- `photo-review + PHOTO_RETAKE -> photo-countdown`
- `photo-review + PHOTO_SKIPPED -> uploading`
- `uploading + UPLOAD_FAILED -> photo-review`
- `uploading + UPLOAD_SUCCEEDED -> shared`

录音确认阶段的双手拇指向上触发 `PHOTO_REQUEST`，不再直接上传。双手拇指向下继续表示放弃并回到自由模式。拍照确认阶段只使用界面按钮，避免现场手势误触。

## 拍照与隐私

- 使用当前 `Game.videoElement` 作为取景源，通过依赖注入传给录音控制器。
- 捕获画面按 4:5 纵向构图裁切，保留中央参与者，避免把两侧围观者作为主体。
- 取景页显示明确的 4:5 边框与“请看镜头”提示；不做人脸识别、身份识别或人脸存储。
- 原始截图仅保存在当前浏览器内存中，完成分享、放弃录音或开始下一次录音时立即释放。
- 上传对象是已经合成、压缩后的 WebP/JPEG 海报，服务端与录音使用相同 token 和 24 小时生命周期。

## 海报版式

画布保持 1080 × 1440，适合手机保存与社交平台分享。

### 主视觉区（0–1040px）

- 照片以 4:5 裁切铺满主视觉区域，使用轻微低饱和青灰色调，保证文字和项目图形可读。
- 左上放置小型项目标识：`WAIC 双手乐队 / LIVE TAKE`。
- 右上放置打卡编号：`PLAYER 0XX`。
- 照片上叠加半透明青色网格、橙红色节奏点和简化的双手追踪线，形成“演奏指纹”。
- 无照片模式使用同样图形系统生成抽象背景，不改变信息结构。

### 票根信息区（1040–1440px）

- 使用浅薄荷色底，延续当前录音卡片配色。
- 左侧为标题与身份信息：
  - `这是我的现场单曲`
  - `你是本场第 0XX 位音乐玩家`
  - `TAKE 0XX · 约 XX 秒`
- 右侧放 250–280px 二维码，并保留足够静区。
- 底部写 `扫码试听与下载 · 24H`，二维码不再占据主视觉。

## 现场确认页

拍照倒计时页在原录音弹窗内切换，不叠加第二层弹窗：

- 背景显示摄像头实时画面。
- 4:5 取景框居中，框外区域压暗。
- 倒计时数字使用大号橙红色数字，分别显示 3、2、1，并在捕获瞬间做一次短白闪。

照片确认页：

- 左侧为大幅照片预览，叠加最小化的演奏指纹预览。
- 右侧为标题“这张作为你的单曲封面吗？”和三项操作。
- 主按钮“使用这张照片”，次按钮“重新拍摄”，文字按钮“无照片继续”。
- 上传中显示“正在制作你的现场单曲封面”，不允许重复提交。

## 云端接口

保持现有音频上传协议，新增二阶段海报上传：

1. `POST /recordings-api/upload` 上传录音并返回 token、编号、过期时间。
2. 浏览器使用返回的分享 URL 和编号生成最终海报。
3. `POST /recordings-api/poster/:token` 上传 `image/webp` 或 `image/jpeg` 海报。
4. `GET /r/poster/:token` 下载海报。

海报最大 2 MB。服务端只允许给现有且未过期的 token 附加一张海报；重复上传覆盖同一 token 的海报。清理任务删除录音元数据时同时删除海报。

## 手机分享页

- 继续把音频播放器放在首屏。
- 增加最终海报缩略图，点击“下载分享海报”直接下载服务端保存的同一张海报，不在手机端重新绘制。
- 海报暂未就绪或旧录音没有海报时隐藏缩略图，并回退到现有本地生成海报逻辑。
- 海报和录音均失效后显示原有过期提示。

## 失败与恢复

- 摄像头未就绪：跳过倒计时，进入无照片确认状态并提示“摄像头暂不可用，可继续生成封面”。
- Canvas 捕获失败：保留录音，允许重拍或无照片继续。
- 录音上传失败：回到照片确认页，保留照片与录音，可重试或下载录音。
- 海报上传失败：自动重试一次；仍失败时仍展示本地海报和音频分享链接，并明确提示手机端暂时只能下载录音。
- 用户放弃：释放照片、海报 Blob、Object URL 和倒计时定时器。

## 性能约束

- 只在拍照时绘制一帧，不启动额外视频编码。
- 捕获图控制在约 900 × 1125；最终海报优先 WebP 质量 0.86，超过 2 MB 时降低到 0.78。
- 海报渲染和上传发生在录音停止后，不影响实时手势识别或音乐播放。
- 不引入持续运行的动画循环或大型前端依赖。

## 验收标准

- 录音确认后不会直接上传，而是进入 3、2、1 拍照流程。
- 能够拍照、重拍、无照片继续并在失败后恢复。
- 最终海报包含照片、编号、二维码、项目识别和演奏指纹，二维码可扫描。
- 原始照片从不离开浏览器；服务端只保存最终海报。
- 手机分享页能够播放录音并下载与现场相同的最终海报。
- 海报和录音在 24 小时后同时失效并被清理。
- 现有录音、下载、邀请码、手势音乐和鼓组功能不受影响。
